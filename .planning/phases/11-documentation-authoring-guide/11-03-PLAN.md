  <files>

 tests/cli_docs_trycmd.rs

  <action>
Implement docs-as-tests with `trycmd`:

1) Add `trycmd = "1"` under `[dev-dependencies]` in `Cargo.toml`.
   - Ensure `assert_cmd` is available in dev-deps (reuse existing if already present). The harness MUST use `assert_cmd::cargo::cargo_bin("cliscrape")` so docs can run `$ cliscrape ...` without relying on PATH.

2) Create `tests/cli_docs_trycmd.rs` with a single test that:
   - Resolves the built binary path via `assert_cmd::cargo::cargo_bin("cliscrape")`.
   - Registers that path for trycmd as the `cliscrape` command (so `$ cliscrape ...` examples are deterministic in CI).
   - Runs trycmd against:
 - `README.md`
 - `docs/*.md`

Example structure (adapt to repo conventions):
```rust
use assert_cmd::cargo::cargo_bin;

#[test]
fn cli_docs() {
    let cliscrape = cargo_bin("cliscrape");
    trycmd::TestCases::new()
        .register_bin("cliscrape", cliscrape)
        .case("README.md")
        .case("docs/*.md");
}
```

3) When authoring docs, prefer `$ cliscrape ...` examples (NOT `cargo run -- ...`). The test harness registration makes `$ cliscrape ...` work in CI/local runs without a global install.

4) Ensure trycmd ignores intentionally non-test blocks:
 - For interactive commands (TUI), mark fenced blocks with `ignore` in the docs content and provide a non-interactive alternative command in a tested block.

5) Keep doc validation safe and repeatable (CI + local):
 - In `tests/cli_docs_trycmd.rs`, delete `target/trycmd` (if it exists) before running trycmd so leftover outputs never make the suite flaky.
 - Require that any doc example that WRITES files uses an explicit output under `target/trycmd/...` (gitignored) and never writes next to tracked fixtures under `docs/_fixtures/`.
 - When authoring docs, prefer unique output paths per guide (e.g. `target/trycmd/migration/minimal.yaml`) to avoid multiple blocks colliding.

Note: Do not create a separate `tests/cmd/*.toml` suite unless needed; keep the first iteration Markdown-driven.
  </action>
  <verify>
Run twice to prove idempotence (no repo pollution / no leftover-output failures):
- `cargo test --test cli_docs_trycmd`
- `cargo test --test cli_docs_trycmd`

If README/docs contain `$ cliscrape ...` blocks, they must execute via the registered `cargo_bin("cliscrape")` path (no requirement to have `cliscrape` installed on PATH).
  </verify>

<task type="auto">
  <name>Task 2: Generate docs/templates.md and enforce determinism in CI</name>
  <files>
docs/templates.md
tests/docs_templates_md.rs
  </files>
  <action>
Create an auto-generated embedded template catalog at `docs/templates.md` and enforce it via an integration test.

Implementation approach (no new runtime features):

1) Add `tests/docs_templates_md.rs` that:
- Builds/runs the repo binary via `assert_cmd`.
- Executes: `cliscrape list-templates --format json`.
- Parses JSON output (array of objects containing at least `name`, `description`, and `source`).
- Derives required catalog fields:
  - `format`: from file extension of `name` (`yaml`/`toml`/`textfsm`/`unknown`).
  - `vendor`: deterministic rule:
    - if `name` contains `/`, vendor is first path segment
    - else vendor is prefix before first `_`
    - fallback `unknown`
- Groups entries by vendor and sorts vendors + entries deterministically by name.
- Renders Markdown that includes:
  - A clear header stating the file is generated and how to regenerate.
  - A short section describing vendor/format derivation rules.
  - For each vendor section: a table with columns: Name, Description, Format, Vendor.

2) The test compares the generated Markdown string to the on-disk `docs/templates.md` and fails if they differ.

3) Provide a supported regen workflow:
- If env var `CLISCRAPE_UPDATE_TEMPLATES_MD=1` is set, the test rewrites `docs/templates.md` with the generated content before failing/passing (so developers can run the test to update).

4) Run the update flow once to create the initial committed `docs/templates.md`.

Keep output stable:
- Strip trailing whitespace.
- Use `\n` newlines.
- Do not include timestamps or build-specific paths.
  </action>
  <verify>
Run:
- `cargo test --test docs_templates_md`
- `CLISCRAPE_UPDATE_TEMPLATES_MD=1 cargo test --test docs_templates_md` (should write/update the file and then pass)
  </verify>
  <done>
docs/templates.md exists, is generated deterministically, and the enforcement test fails on drift.
  </done>
</task>
