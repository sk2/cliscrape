    1. Action model:
       - Extend `src/engine/types.rs` so record actions can represent `Clearall` and `Error` distinctly.
         - Acceptable approach: keep `Action` but add variants `ClearAll` and `Error`, and continue treating `Action::Next` as `NoRecord` when used as a record action.
    2. Loader parsing:
       - In `src/template/loader.rs`, parse record actions:
         - `Clear` -> `Clear`
         - `Clearall` -> `ClearAll`
         - `Error` -> `Error`

       - In `src/engine/records.rs`, add helpers so `Clear` clears only non-Filldown values and `ClearAll` clears everything.
       - In `src/engine/fsm.rs`, implement:
         - `Clear` -> clear non-Filldown
         - `ClearAll` -> clear all
         - `Error` -> immediately return `Err(...)` (fail fast); ensure no results are returned.
  </action>

<task type="auto">
  <name>Task 2: Implement Required + Filldown record semantics (including record gating + post-record clearing)</name>
  <files>src/engine/records.rs, src/engine/fsm.rs, src/template/loader.rs, tests/textfsm_compat.rs, tests/fixtures/textfsm/test_required.textfsm, tests/fixtures/textfsm/test_filldown.textfsm</files>
  <action>
    Implement TextFSM's `Value Required` and `Value Filldown` behaviors explicitly and lock them down with fixture-backed tests.

    1. Loader/IR:
       - Ensure `src/template/loader.rs` correctly sets `Value.required` and `Value.filldown` when parsing `Value Required ...` and `Value Filldown ...` lines.
       - Keep parsing strict: unknown value flags still follow the warn+ignore behavior from Task 3.

    2. Record emission gating (Required):
       - When a rule triggers a record action (`-> Record` or equivalent), only emit a row if *all* `required=true` values are present and non-empty.
       - If required values are missing:
         - Do NOT emit a partial row.
         - Still apply the same post-record clearing behavior described below (so subsequent lines behave deterministically).

    3. Post-record clearing semantics (Filldown interaction):
       - After a successful record emission, clear all non-Filldown values; preserve Filldown values for subsequent records.
       - After a failed record emission due to missing Required values, apply the same clearing rule (clear non-Filldown, preserve Filldown) to match typical TextFSM expectations and avoid stale non-filldown values.
       - Ensure Task 1's `Clear` / `Clearall` behavior remains correct with these semantics:
         - `Clear` clears only non-Filldown values.
         - `Clearall` clears everything.

    4. Fixtures (create/ensure exact contents):
        - Ensure `tests/fixtures/textfsm/test_required.textfsm` exists with:
          ```textfsm
          Value Required ID (\d+)
          Value DATA (\w+)

         Start
           ^ID ${ID}
           ^Data ${DATA}
            ^Emit -> Record
          ```
        - Ensure `tests/fixtures/textfsm/test_filldown.textfsm` exists with:
          ```textfsm
          Value Filldown CHASSIS (\S+)
          Value SLOT (\d+)

         Start
           ^Chassis ${CHASSIS}
           ^Slot ${SLOT} -> Record
          ```

    5. Tests (use those fixtures):
        - In `tests/textfsm_compat.rs`, add tests that load `tests/fixtures/textfsm/test_required.textfsm` and `tests/fixtures/textfsm/test_filldown.textfsm`:
          - Required: input containing `Data ...` and `Emit` without `ID` produces *zero* rows; input with both `ID` and `Data` produces one row.
          - Filldown: input with one `Chassis ...` and multiple `Slot ...` lines emits multiple rows where `CHASSIS` is repeated (filldown) and `SLOT` changes.
  </action>
  <verify>`cargo test -q` passes and `tests/textfsm_compat.rs` includes assertions for both `Value Required` and `Value Filldown` using `tests/fixtures/textfsm/test_required.textfsm` and `tests/fixtures/textfsm/test_filldown.textfsm`.</verify>
  <done>`Value Required` suppresses record emission when missing; `Filldown` values persist across records; after record attempts, non-filldown values clear deterministically while filldown values persist until Clearall.</done>
</task>
