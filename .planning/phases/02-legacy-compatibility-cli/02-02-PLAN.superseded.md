<task type="auto">
  <name>Task 1: Enforce exit-code + error-format contract and deterministic template resolution</name>
  <files>src/main.rs</files>
  <action>
    Update `src/main.rs` to become the Phase-2 adapter layer (engine remains pure):
    1. Pre-scan raw argv for `--error-format` *before* clap parsing so clap/usage errors can still be formatted as JSON even when clap fails to parse.
       - Scan `std::env::args_os()` for either `--error-format <human|json>` or `--error-format=<human|json>`.
       - Accept the flag anywhere in argv (ordering must not matter):
         - `cliscrape parse --wat --error-format json`
         - `cliscrape --wat --error-format json`
       - If the flag is present but value is missing/invalid, fall back to `human`.
       - Use the pre-scanned selection as the formatting choice for *clap errors only*.
    2. Parse args with `Cli::try_parse()` (NOT `Cli::parse()`), so clap parse failures can be handled.
       - On clap error: print to stderr using the pre-scanned error format and `exit(1)`.
    3. Implement `--error-format human|json` behavior for runtime errors:
        - Default human-readable errors on stderr.
        - JSON error envelope on stderr when requested (e.g. `{ok:false, message, kind, context?}`).
        - Ensure all runtime errors (template load, input read, parse) also flow through this path.
    4. Deterministic template selection (no template-root/search path):
        - If `--template` value is an existing path: use it.
        - Else treat it as an identifier and search *current working directory only* for `<id>.textfsm|yaml|yml|toml` (restricted by `--template-format` when not `auto`).
        - 0 matches: error telling user to pass a path.
        - >1 matches: ambiguity error; instruct user to pass `--template-format` or a full path.
    5. Load templates via library:
        - Use `FsmParser::from_file_with_format` when format override is set.
        - Prefer the new warning-returning loader API from plan 02-04 and collect warnings for stderr.
  </action>
  <verify>
    1. `cargo test -q` passes and `cargo run -- parse --help` still works.
    2. Clap error exit code is `1`: `cargo run -- parse` exits `1` (missing required `--template`).
    3. Clap JSON error is valid JSON and `ok:false`:
       - `cargo run -- --error-format json parse 2>err.json ; test $? -eq 1`
       - `python -c 'import json; d=json.load(open("err.json")); assert d.get("ok") is False'`
    3a. Clap JSON error selection is order-independent (unknown flag before/at root):
        - `cargo run -- parse --wat --error-format json 2>err.json ; test $? -eq 1`
        - `python -c 'import json; json.load(open("err.json"))'`
        - `cargo run -- --wat --error-format json parse 2>err.json ; test $? -eq 1`
        - `python -c 'import json; json.load(open("err.json"))'`
    4. Runtime JSON error is also valid JSON and `ok:false`:
       - `cargo run -- --error-format json parse -t does_not_exist 2>err.json ; test $? -eq 1`
       - `python -c 'import json; d=json.load(open("err.json")); assert d.get("ok") is False'`
  </verify>
  <done>All failures (including clap arg errors) exit `1` and print errors to stderr in the selected format; JSON errors are parseable JSON with `ok:false`; template identifier ambiguity is an error.</done>

<task type="auto">
  <name>Task 2: Implement multi-input resolution (stdin + explicit + glob) with fail-fast/no-partial-output</name>
  <files>Cargo.toml, src/main.rs, src/transcript/mod.rs, src/transcript/ios_prompt.rs</files>
  <action>
    1. Add dependencies:
       - `glob = "0.3"` for explicit `--input-glob` expansion.
       - `strip-ansi-escapes = "0.2"` for transcript normalization.
    2. Resolve inputs deterministically:
       - Gather explicit file paths from positional `inputs...` and/or `--input`.
       - Expand each `--input-glob` pattern using the `glob` crate; error if a glob matches zero files.
       - Include stdin when `--stdin` is set.
       - Also include stdin by default when *no explicit inputs were provided* AND stdin is not a TTY (piped input).
       - Define a stable source ordering for combined runs:
         - Process file inputs first (sorted paths, de-duped), then process stdin last when included.
         - Preserve per-source row order; overall output row order is the concatenation of sources in that stable order.
       - Sort paths deterministically and de-duplicate exact duplicates.
       - Error if final input set is empty.
    3. Transcript cleanup is smart-by-default:
       - In `src/transcript/mod.rs`, add a new helper used by the CLI that:
         - strips ANSI escapes;
         - applies IOS prompt/echo segmentation (`preprocess_ios_transcript`);
         - returns `(blocks, warnings)` so the CLI can emit warnings to stderr.
    4. Parse pipeline:
       - For each input source (stdin or file), read fully to string, preprocess to blocks, then parse each block.
       - Fail fast on the first error.
       - Do not emit any records to stdout until *all inputs* parse successfully.
       - Accumulate results in memory and serialize once at the end.
  </action>
  <verify>
    1. `echo "x" | cargo run -- parse -t some_template --error-format human` exits `1` with a human error to stderr (template not found).
    2. Verify stdin+file combining works using repo fixtures:
       - `printf 'Interface GigabitEthernet3 is up, line protocol is up\n  Internet address is 10.1.1.3/24\n' | cargo run -- parse -t examples/cisco_show_interfaces.textfsm examples/output.txt --stdin --format json | python -c 'import json,sys; a=json.load(sys.stdin); assert len(a)==3'`
    3. `cargo test -q` passes.
  </verify>
  <done>Inputs can be combined (stdin + files + globs) with deterministic ordering; any failure results in exit `1` and no stdout records.</done>
</task>
