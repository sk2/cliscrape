<task type="auto">
  <name>Task 1: Add tracing dispatch variants to throughput benchmark</name>
  <files>benches/throughput.rs</files>
  <action>
Update `benches/throughput.rs` to benchmark the same parse workload under multiple tracing configurations using `tracing::dispatcher::with_default`:

- Baseline: no custom dispatch (current behavior).
- Default subscriber (matches production default behavior from plan 10-01 when `RUST_LOG` is unset and verbosity=0): a `tracing_subscriber::fmt()` subscriber with filter `warn` and writer set to `std::io::sink`.
- Disabled subscriber: a `tracing_subscriber::fmt()` subscriber with an EnvFilter/LevelFilter that effectively disables events (OFF), writer set to `std::io::sink`.
- Enabled text subscriber: max level TRACE (or filter `cliscrape=trace`), writer `std::io::sink`.
- Enabled JSON subscriber: same max level, `.json()` enabled, writer `std::io::sink`.

Make each configuration a separate Criterion bench function name, e.g.:
- `parse 200k lines (baseline)`
- `parse 200k lines (tracing default)`
- `parse 200k lines (tracing off)`
- `parse 200k lines (tracing text)`
- `parse 200k lines (tracing json)`

Avoid printing during benches; keep writers to sink to measure formatting overhead without IO.
For each variant that installs a subscriber, construct the subscriber + `Dispatch` once per bench function and scope `tracing::dispatcher::with_default(&dispatch, || b.iter(|| ...))` around the entire `b.iter` call (do NOT call `with_default` inside the `b.iter` closure).
  </action>
  <verify>
Run `cargo bench --bench throughput`.
  </verify>
  <done>
Throughput bench includes baseline + tracing variants and runs successfully.
  </done>
</task>
